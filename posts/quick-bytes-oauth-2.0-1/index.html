What is OAuth?#

OAuth (“Open Authorization”) is a standard authorization protocol designed to allow a website or application to access resources hosted by other web apps on behalf of a user. It allows consented access and restricts actions of what a client app can perform on user resources on behalf of the user, without ever sharing the user’s credentials.

⚠️ Did you notice it’s an authorization protocol and not an authentication protocol?

  • Authentication verifies who a user is, confirming their identity. It answers, “Are you who you claim to be?”.
  • Authorization determines what a verified user or system can do, defining their permissions or access rights (e.g., read, write, admin). It answers, “What are you allowed to do?”

Authentication is about identity verification; authorization is about access control. Authentication typically happens first, followed by authorization to enforce appropriate privileges.

Access tokens#

OAuth 2.0 uses Access Tokens to authorize a client (third party application) to access a user’s resource on their behalf. An Access Token is a piece of data that represents the authorization to access resources on behalf of the end-user. OAuth 2.0 doesn‚Äôt define a specific format for Access Tokens. Roles#

OAuth defines four roles

  1. Resource owner: An entity capable of granting access to a protected resource. When the resource owner is a person, it is referred to as an end-user. Resource owner is the user or system that owns the protected resources and can grant access to them.

    Examples:

    • A user with a google account is a resource owner.
    • A user with a github account is a resource owner.
    • A user with a reddit account is a resource owner.

  2. Resource server: The server hosting the protected resources, capable of accepting and responding to protected resource requests using access tokens. Example resource servers:

    • 🗓️ https://www.googleapis.com/calendar/v3,
    • ✉️https://gmail.googleapis.com,
    • 📷https://photoslibrary.googleapis.com,
    • ✨ https://generativelanguage.googleapis.com

  3. Client: An application making protected resource requests on behalf of the resource owner and with its authorization. To access resources, the Client must hold the appropriate Access Token. Example:

    • Notion mail is a client to access gmail.
    • Apple calendar is a client to access calendars from various resource servers of Google, Yahoo etc.

  4. Authorization server: The server issuing access tokens to the client after successfully authenticating the resource owner and obtaining authorization. Example:

    • https://oauth2.googleapis.com/token is an authorization server that can issue tokens to access to various services Google access tokens
    • https://github.com/login/oauth/access_token is an authorization server that can issue tokens to access Github

    NOTE: The authorization server may be the same server as the resource server or a separate entity

OAuth grant types (flows)#

📝 ‚Äúgrant type‚Äù refers to the way an application gets an access token. Each grant type is optimized for a particular use case, whether that‚Äôs a web app, a native app, a device without the ability to launch a web browser, or server-to-server applications.

Authorization code#

  • Requires launching a web browser (user-agent) to complete the workflow

  • Suitable workflow for webapps with a backend

  • User (Resource owner) clicks on a button/link on a third party app to access their content from their resource provider (Example: Gmail)

  • The link redirects them to Resource provider’s authorization server page (consent form) with the list of scopes (permissions) required

  • User authenticates with authorization server and consents to allowing the third party to access resources from resource server on their behalf

  • On receiving a consent, the authorization server returns a response to the user agent (browser) with an HTTP status 302 (redirect) and a Location specified by the third party client along with an authorization code. This location is typically where third party backend handles the rest of the workflow

  • Third party backend uses the short lived authorization code, it’s client id, client secret to request an access token from authorization server. Client secret ensures that registered client is the only entity that is allowed to request access tokens. Unregistered clients or malicious actors cannot request access tokens since they don’t have the client secret. Client secret is never exposed to the front end (browser)

  • On verifying the authorization code and client parameters, authorization server creates a short-lived access token and returns it to the backend of requesting client

  • Client backend uses the access token to access user’s resources from Resource server

  • Example flow
    • Notion register themselves as clients (with a redirect URL) on Google to receive their client id and client secret
    • User: “Hey Notion!, I want to see my gmail emails here in your app”
    • Notion: “Sure, click this link that redirects you to google account authentication and authorization” (User clicks the link to open a browser window with Google authentication)
    • User: “Hello google! Here’s my email and password, please verify and allow me to grant read access to Notion”
    • Google: “Sure, let me check your credentials. Ah, you’re a legit user. I will let you in. Oh btw, Notion wants a scope to read your email, profile. Do you want to grant those permissions to Notion? You can revoke it anytime you want”
    • User: “Yup” (Google generates an authorization code and redirects user’s browser to make a call to redirect URL specified by Notion during registration)
    • Notion backend: “Great! Now we have an authorization code. Hey google, here’s my client id, client secret and the authorization code. Please give me an access token with the requested scopes”
    • Google: “Yes, I can confirm you’re a registered client with a valid authorization code. Here’s your access token. Remember it expires after a while”
    • Notion backend: “Awesome! Thanks”
    • Notion backend: “Hello Gmail server! Here’s my access token, I need to read emails of this user”
    • Gmail server: “Let me verify that token first. Ah great! That’s valid and you have a valid scope. Here are the user’s emails, enjoy!”
    • Notion: “Hello user, here are your emails. Lets use notion’s awesome features on these emails now, shall we?"
